Day 1–2 — Activate what you already pay for
Zero cost · Zero install · Immediate visibility · ~3 hours total
1.1
Enable Defender for Cloud — all plans
Azure Portal → Defender for Cloud → Environment Settings → enable Defender for Containers, Defender for Servers (Kafka VMs + Jump VMs), Defender for Storage, Defender for Key Vault. Check your secure score immediately — this is your baseline number.
already licensed~45 min
1.2
Enable Defender for Containers → AKS integration
This installs the Defender sensor as a DaemonSet in AKS automatically. Gives you runtime alerts, image vulnerability scanning from ACR, and K8s audit log analysis — without touching anything else. In Azure Portal: Defender for Cloud → Workload Protections → Containers.
already licensed~20 mindo this first
1.3
Review Defender recommendations — triage top 10
Defender for Cloud → Recommendations → filter by severity "High". You'll see a list of specific misconfigs across all your resources. Don't fix everything — just read and categorise. Note what's in networking (App GW, Firewall rules), Key Vault, and AKS. This becomes your initial risk register.
already licensed~1 hour
1.4
Enable ACR vulnerability scanning
Defender for Containers automatically scans images pushed to your Azure Container Registry. Go to ACR → Microsoft Defender for Cloud tab and confirm it's active. All future image pushes will be scanned. Historical images get scanned within 24 hrs.
already licensed~10 min
2
Week 1 — Scan your K8s and Azure posture
Free open-source tools · One-time scans · Find the unknown unknowns
2.1
Run Kubescape against AKS — get a K8s risk score
Install: curl -s https://raw.githubusercontent.com/kubescape/kubescape/master/install.sh | /bin/bash then run kubescape scan --enable-host-scan --format html --output aks-report.html against your cluster. You'll get a risk score (0–100), failed controls, and exact namespaces/workloads affected. This is your K8s baseline.
free~30 min setup + 10 min scanmost urgent
2.2
Run kube-bench for CIS K8s benchmark
Run as a K8s Job: kubectl apply -f https://raw.githubusercontent.com/aquasecurity/kube-bench/main/job-aks.yaml then kubectl logs job/kube-bench. Gives PASS/FAIL on node configuration, API server settings, etcd. AKS manages master nodes so some checks will be N/A — that's expected.
free~20 min
2.3
Run Prowler against Azure subscription
Install: pip install prowler. Auth: az login. Run: prowler azure --az-cli-auth --output-formats html json --output-directory ./prowler-out. Takes 20–40 min. You'll get findings across all 21 services — Storage, Key Vault, Networking, PostgreSQL, Redis, Entra ID. Filter by CRITICAL and HIGH first.
free~1 hour total
2.4
Run AzureHound → BloodHound CE for identity mapping
Install BloodHound CE via Docker: docker compose up from their repo. Run AzureHound collector to dump Entra ID + RBAC data. Import into BloodHound. Look specifically for: service principals with Key Vault access, Contributor/Owner role assignments, any identity that can reach your PostgreSQL or Redis. This will likely surprise you.
free~2 hours first time
3
Week 2 — Add runtime detection + shift-left in CI/CD
Falco in K8s · Trivy in pipeline · Gitleaks on repos
3.1
Deploy Falco in AKS as DaemonSet
Install via Helm: helm repo add falcosecurity https://falcosecurity.github.io/charts && helm install falco falcosecurity/falco --set falco.grpc.enabled=true -n falco --create-namespace. Configure a webhook to send alerts to Teams or email. Default rules already catch: shell spawned in container, unexpected outbound network, write to /etc, privilege escalation. Test it by exec-ing into a pod — you should get an alert within seconds.
free~2 hours
3.2
Add Trivy to your CI/CD pipeline
In Azure DevOps or GitHub Actions, add a Trivy scan step after your Docker build. Block deploys with CRITICAL CVEs. Also add trivy config . to scan your Helm charts and K8s YAML at the same time. Simple YAML snippet: aquasecurity/trivy-action GitHub Action or aquasec/trivy Azure DevOps task.
free~1 hour per pipeline
3.3
Run Gitleaks + TruffleHog on all repos (one-time history scan)
One-time deep scan: trufflehog git <repo-url> on every repo. This finds secrets committed in the past — Azure storage keys, connection strings, Entra client secrets. Then add Gitleaks as a pre-commit hook and CI gate going forward.
free~1 houroften finds real leaks
3.4
Run ScubaGear on O365
PowerShell: Install-Module -Name ScubaGear; Invoke-SCuBA. Checks MFA enforcement, external sharing in SharePoint/Teams, DKIM/DMARC on your domain, legacy auth protocols. Generates an HTML report. Fix any HIGH findings — especially legacy auth (common phishing vector).
free~1 hour
4
Week 3–4 — Build the unified dashboard (DefectDojo)
Centralise all findings · assign · track · close · build recurring schedules
4.1
Deploy DefectDojo on a small Azure VM or K8s namespace
Docker Compose: git clone https://github.com/DefectDojo/django-DefectDojo && docker compose up. Or deploy the Helm chart into a dedicated namespace in AKS. A Standard_B2s VM (~$30/mo) is enough. This becomes your permanent security operations hub.
free software~3 hours setup
4.2
Import all Phase 2+3 findings into DefectDojo
DefectDojo has built-in parsers for Prowler JSON, Kubescape JSON, Trivy JSON, Checkov JSON, and Gitleaks. Create a Product per system (e.g. "AKS Cluster", "Azure Infra", "Container Images"). Import each tool's output. DefectDojo deduplicates across scans and tracks findings over time — so re-running Prowler next week shows you what's new vs what's the same.
free~2 hours
4.3
Set up recurring scans on a schedule
Prowler + Kubescape: cron job weekly (or nightly if you want). Trivy image scan: every CI/CD pipeline run. Falco: always-on. Gitleaks: every commit. ScubaGear: monthly. Store all JSON outputs in an Azure Storage Blob and import to DefectDojo automatically via its API. This closes the loop — you now have continuous visibility without a commercial platform.
free~2 hours to automate
