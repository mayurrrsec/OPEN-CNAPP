01 What you have now — honest assessment

The core problem

The dark theme, bullet-list plugin manager, plain-text pentest runner, and empty charts make it look like a CLI tool with a web skin. Users can't tell what's happening, what to do next, or how bad their security posture is. The reference products (AccuKnox, Wiz, Orca) solve this with visual hierarchy, rich widgets, and clear user flows.

Findings: table with filters — functional but bare
Pentest: two text inputs + Run button + "null / []" output

The backend is largely done — adapters, normalizers, ingest endpoints are all in place. This is a frontend-heavy project. Estimated: 8–10 weeks of focused frontend work by one developer, or 4–5 weeks with two.

02 What industry-standard looks like

Based on AccuKnox CNAPP dashboard, Orca Security, and Wiz — here are the non-negotiable elements that define a professional CNAPP UI:

Risk score / secure score
A 0–100 number with a visual gauge. The single most important number on the platform. Drives the whole conversation.
Multi-cloud header bar
Add Cloud · Add Cluster · Add Registry buttons in the top right. Users immediately see how to onboard resources.
Findings by cloud provider
Table or bar per cloud (AWS, Azure, GCP) showing total findings with C/H/M/L severity pills. Essential for multi-cloud teams.
Customizable widgets
Users can add/remove widgets from a catalogue of 30+. AccuKnox has 117 widgets across CSPM, KSPM, Runtime, Assets, Tickets domains.
Namespace-level K8s view
K8s clusters → namespaces → workloads. Severity per namespace. Alerts per pod. This is the #1 K8s-specific feature users expect.
Compliance framework heatmap
Control domains × pass/fail/partial. Color-coded grid. Click any cell → drilldown to specific findings. Not just counts.
Attack path visualization
Graph where nodes are resources, edges are risk relationships. Critical paths highlighted in red. Wiz calls this Security Graph.
Real-time alert feed
Falco/runtime events streaming in live. Severity color-coded. Click to expand. Filter by cluster/namespace/severity.
Image risk assessment
Container images by vulnerability count. Critical images flagged. Links to registry scan queue. Essential for container-heavy teams.
Light theme is mandatory

AccuKnox, Wiz, and Orca all default to light theme with dark mode available. Enterprise security teams work in well-lit offices and often screen-share dashboards in presentations. A dark-only dashboard reads as a "developer tool" not a "security product." Switch your default to light theme and offer a toggle.

04 Design system overhaul

Do this first — before any new features

Build a proper CSS design token system and component library first. Every new screen you build without this will need to be rebuilt later. One week on design system = saves 3 weeks of inconsistent rework.

Switch default to light theme. Keep dark as an option. Define semantic tokens:

07 Onboarding flows — critical missing piece

Why this matters most

Your dashboard currently shows "No findings yet" with no clear call to action. A new user has no idea how to add their cloud account, connect a cluster, or trigger their first scan. This is the #1 UX problem causing the product to feel like a prototype.

Cloud onboarding wizard (4 steps)

Step 1: Choose cloud provider
  [Azure] [AWS] [GCP] [On-prem]

Step 2: Enter credentials (form changes per cloud)
  Azure: Subscription ID · Tenant ID · Client ID · Client Secret
  AWS:   Access Key ID · Secret Access Key  (or IAM Role ARN)
  GCP:   Upload service account JSON

Step 3: Test connection
  [Testing...] → [✓ Connected — 47 resources found]
  or [✗ Auth failed — check credentials]

Step 4: Choose first scan
  [Run Prowler now (CSPM)] [Schedule for tonight] [Skip for now]

Cluster onboarding wizard (3 steps)

Step 1: Name your cluster + choose cloud
  Cluster name: [aks-prod-eastus]
  Cloud: [Azure AKS ▼]

Step 2: Install agent
  Run this command on your cluster:
  kubectl apply -f https://opencnapp.local/api/agent/install/{token}

Step 3: Verify
  [Checking connection...] → [✓ Cluster connected — 12 nodes, 47 pods]

First-run experience

When the database is empty, show a full-page onboarding prompt instead of "No findings yet." Three clear paths: "Connect a cloud account" · "Connect a Kubernetes cluster" · "Ingest from CI/CD pipeline." Each path has a single CTA button leading to the wizard.

11 Agent handoff spec — what to give Cursor / Codex

The problem with the previous agent output

Cursor/Codex built a functional prototype but without a strong design spec. The agents defaulted to the simplest possible implementation of each screen. The solution is to give the next agent extremely specific UI specs with references to AccuKnox screenshots, explicit component names, and pixel-level detail.

What to include in the next agent prompt

Reference screenshots
Attach the AccuKnox screenshots (images 10–12 in this conversation) as visual references. Say "Match this visual quality and layout."
Design token file
Give the agent a complete tokens.css file with all severity colors, cloud colors, domain colors, and spacing. Agents follow token files precisely.
Component names
Name every component explicitly: SeverityBadge, CloudBadge, PluginCard, FindingRow, DrawerPanel, WizardStepper, ComplianceHeatmap, RiskGauge.
One page per task
Don't ask for all pages at once. One agent run = one page. "Build the Overview page with these exact widgets: [list]." Quality is far higher this way.
shadcn/ui explicitly
Tell the agent to use shadcn/ui components by name. "Use shadcn Card, Dialog, Select, Badge, Tabs, Button. Use Lucide React for icons."
Explicit API endpoints
Give each page spec its exact API calls: "Fetch GET /dashboard/summary and display the result.severity_counts in a SeverityDonut chart."

Build the [PAGE NAME] page for OpenCNAPP (React + TypeScript + Tailwind + shadcn/ui).

Visual reference: match the AccuKnox CNAPP dashboard quality shown in [screenshots].
