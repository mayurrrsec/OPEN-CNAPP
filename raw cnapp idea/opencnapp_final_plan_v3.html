01 All open questions answered

Q Is this architecture scalable for open-source use?
Yes — by design. Adding a tool = 2 files (YAML + Python adapter), no core changes. Scaling = add Celery workers. Phase 1: Docker Compose on a laptop. Phase 4: Helm chart on K8s with HPA. The plugin system is the same code at all scales.
Q Will the architecture support multiple deployment options later?
All four options are designed in from day 1. Option A: docker compose up (solo). Option B: Ubuntu VM + nginx (team). Option C: TLS + SSO/LDAP (enterprise on-prem). Option D: Helm chart on K8s (K8s-first teams). Same Docker images, same API, just different compose profiles.
Q Will we give connectors for all clouds?
Yes — Azure, AWS, GCP, K8s, on-prem/VM, Git/SaaS, O365, Google Workspace. Each connector is a Python class with a standard interface. Adding Alibaba Cloud = one new connector file. See Section 4 for the complete connector registry.
Q If Defender for Cloud or AWS Security Hub is enabled — will findings show in our dashboard?
Yes — via optional native security ingest adapters. If the user has Defender for Cloud, we pull its recommendations via Azure REST API and normalize them into our findings schema. Deduplication by (resource_id + check_id) prevents duplicates when Prowler and Defender find the same thing. Findings show a "sources" field. See Section 5.
Q Will the dashboard be industry standard?
Yes — modelled after Wiz, Orca, AccuKnox, and Sysdig. Includes: risk score (0-100), attack path graph (D3.js), findings table with full filters, severity/domain/trend charts (Recharts), compliance heatmap (CIS/NIST/PCI/SOC2/ISO27001), real-time alert feed (WebSocket). See Section 6 for full mockup.
Q What about SonarQube, ZAP, SBOM, and other CI/CD tools?
All supported via CI/CD Connector system in Phase 5. SonarQube: pull mode (API) or push mode (webhook). ZAP: POST JSON report to /ingest/zap from pipeline. SBOM: CycloneDX and SPDX format ingest. Snyk, Semgrep, Bandit, Dependency-Check, Fortify, Veracode also planned. See Section 8.
Q What about ThreatMapper (Deepfence)?
It's a strong open-source CNAPP we integrate with rather than compete with. ThreatMapper is an optional plugin: OpenCNAPP ingests its findings via ThreatMapper's API. In v2, we implement our own ThreatGraph in D3.js using our own attack path data. See Section 2 for full analysis.

02 ThreatMapper — what it is and how it fits

ThreatMapper is an open-source, cloud-native application protection platform (CNAPP) built by Deepfence. It scans for vulnerabilities, malware, compliance misconfigurations, and exposed secrets, then prioritizes alerts by exploitability. It uses a combination of agent-based inspection and agent-less monitoring across VMs, containers, Kubernetes, and serverless.

Key feature: ThreatGraph

ThreatMapper's ThreatGraph visualizes attack paths — which threats present the greatest risk to your applications based on runtime context like network flows. This is the same concept as Wiz's Security Graph. It's one of the most useful features in any open-source CNAPP.

Decision: integrate, don't compete

ThreatMapper is an optional plugin. Users who want its ThreatGraph can run it as a sidecar Docker Compose profile. OpenCNAPP ingests its findings via its REST API. In v2, we implement our own lightweight ThreatGraph using D3.js force-directed graph.

# plugins/threatmapper/plugin.yaml
name: threatmapper
display_name: "ThreatMapper (Deepfence)"
domain: cnapp
run_mode: api_ingest          # pulls from ThreatMapper's REST API
endpoint: "http://threatmapper:8080"
normalizer: adapters.threatmapper.normalize
dashboard_url: "http://threatmapper:8080"  # link-out to native UI
optional: true
enabled_by_default: false

Credential security

All credentials encrypted with AES-256. Key derived from SECRET_KEY in .env. Never logged. Mounted as env vars into scanner containers at runtime only — destroyed after scan.

Deduplication

When Prowler and Defender for Cloud both find "Storage account allows public blob access" on the same resource, OpenCNAPP deduplicates by fingerprint = SHA256(resource_id + check_id) and shows the finding once with sources: ["prowler", "defender_for_cloud"]. No duplicate noise.

09 Common findings schema

The single most important design decision. Every tool adapter maps to this shape. Build this first.

CREATE TABLE findings (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tool            VARCHAR(50) NOT NULL,   -- "prowler"|"trivy"|"falco"|"sonarqube"|"zap"...
  scan_id         UUID NOT NULL,
  source          VARCHAR(30) NOT NULL,   -- "scheduled"|"ci_ingest"|"on_demand"|"webhook"|"native_ingest"
  domain          VARCHAR(30) NOT NULL,   -- "cspm"|"kspm"|"cwpp"|"ciem"|"image-sec"|"iac"|"secrets"|"sast"|"dast"|"pentest"
  severity        VARCHAR(10) NOT NULL,   -- "CRITICAL"|"HIGH"|"MEDIUM"|"LOW"|"INFO"
  cvss_score      FLOAT,
  cve_id          VARCHAR(30),
  cloud_provider  VARCHAR(20),               -- "azure"|"aws"|"gcp"|"k8s"|"onprem"|null
  account_id      VARCHAR(200),
  region          VARCHAR(100),
  resource_type   VARCHAR(200),
  resource_id     TEXT,
  resource_name   VARCHAR(500),
  namespace       VARCHAR(200),              -- K8s namespace
  check_id        VARCHAR(200),              -- tool's native check ID
  title           TEXT NOT NULL,
  description     TEXT,
  remediation     TEXT,
  references      JSONB,
  compliance_tags VARCHAR(50)[],             -- ["cis-azure-1.4","pci-dss","nist-800-53"]
  status          VARCHAR(30) DEFAULT 'open',
  assignee        VARCHAR(200),
  ticket_url      TEXT,
  fingerprint     VARCHAR(64),               -- SHA256(tool+check_id+resource_id) for dedup
  sources         VARCHAR(50)[],             -- ["prowler","defender_for_cloud"] if deduplicated
  first_seen      TIMESTAMPTZ DEFAULT NOW(),
  last_seen       TIMESTAMPTZ DEFAULT NOW(),
  scanned_at      TIMESTAMPTZ NOT NULL,
  raw_finding     JSONB
);

10 Scalability design

Phase 1 — Solo (today)
docker compose up on laptop or $20/mo VM. ~400MB RAM. All services on one machine.
Phase 2 — Team (2-10 people)
Same compose + nginx + 3x workers. ~1GB RAM. $20-40/mo VM. Internal team access.
Phase 3 — More scans
docker compose up --scale worker=5. 5 parallel scan workers. Zero code change.
Phase 4 — Enterprise
Helm chart on K8s. HPA on workers. Managed Postgres. Same API, same Docker images.
What never needs re-architecture

Adding a new tool (YAML + adapter), new cloud (connector class), new CI source (ingest adapter), more findings (Postgres + JSONB handles millions), more users (stateless FastAPI, JWT, horizontal scale).
