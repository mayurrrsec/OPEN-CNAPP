Complete architecture, installation guide, and implementation roadmap for a unified cloud-native security platform on Azure + AKS. Everything Wiz and Orca do — built with open-source tools, running on your own infrastructure.

01

What is CNAPP and what domains does it cover

CNAPP = Cloud-Native Application Protection Platform. It is an umbrella that unifies multiple security disciplines into one view.

The commercial secret: Wiz, Orca, and AccuKnox don't have magic algorithms. They run the same open-source engines in the background (Prowler, Falco, Trivy, etc.), normalize their JSON output into one schema, and present it in a React dashboard. You can build the same thing.

CSPMCloud Security Posture MgmtAre my Azure resources configured securely? Storage public? Key Vault accessible?Critical
KSPMKubernetes Security Posture MgmtAre my AKS workloads following best practices? RBAC correct? Pods running as root?Critical

Every open-source tool — what it does, how it runs

CSPM — Azure cloud posture Azure

Checks whether Azure services are configured securely. Runs on a schedule, reads Azure APIs.

Prowler 400+ Azure checks. VNet, Firewall, Key Vault, Storage, Entra ID, PostgreSQL, Redis. Outputs JSON/HTML/CSV. Primary CSPM tool.
ScoutSuite Multi-cloud. Good Azure coverage. HTML report per scan. Good for periodic audit snapshots alongside Prowler.
Defender for Cloud ✓ You already have it. Azure-native secure score. Enable all plans. Free tier useful but limited.
KSPM — Kubernetes posture AKS

Checks AKS cluster config, RBAC, pod specs, network policies against CIS benchmarks.

Kubescape CIS K8s benchmark, NSA hardening, RBAC, pod security. Has free ARMO cloud UI. Best single KSPM tool for AKS.
Polaris Deploys in-cluster. Checks pods for root containers, missing resource limits, missing probes, :latest tags. Built-in web UI.
kube-bench ✓ You already run this. CIS K8s benchmark against AKS node config. CLI only. Run after AKS changes.
CWPP — Runtime protection Real-time

Monitors running containers and VMs in real time for malicious behaviour. Most important security tool.

Falco Deployed as DaemonSet. Syscall-level monitoring. Alerts on shell spawn in container, /etc writes, suspicious outbound. Webhooks to Slack/Teams. Deploy this first.
Tracee eBPF-based alternative to Falco by Aqua Security. Better on newer AKS kernels. Use one or the other, not both.
CIEM — Identity & entitlements Entra ID

Maps who can access what in Azure — service principals, managed identities, attack paths.

AzureHound Collects Entra ID and Azure RBAC data. Feeds into BloodHound graph. Runs as a one-off or scheduled job.
BloodHound CE Free Community Edition. Full graph UI showing attack paths from any identity to Key Vault / subscriptions. Best CIEM tool available.
Container image scanning CI/CD

Scans Docker images for CVEs, secrets baked in, and misconfigurations before they reach AKS.

Trivy ✓ Already in CI/CD. Also run in cluster mode: trivy k8s --report summary. Scans ACR images, Helm charts, Dockerfiles. Industry standard.
Grype + Syft ✓ Already running. Syft generates SBOM, Grype scans it for CVEs. Keep both.
IaC security + Secrets detection Shift-left

Catches misconfigs and secrets before code is even deployed. Lives in CI/CD pipelines.

Checkov Scans Terraform, Bicep, ARM, Helm, K8s YAML, Dockerfile. Run as CI/CD gate. JSON/JUnit output.
Gitleaks Scans git repos for Azure connection strings, storage keys, Entra client secrets. Run as pre-commit hook + CI gate.
TruffleHog Deep git history scan. Finds secrets committed years ago and deleted. Run once on all repos, then periodically.
SSPM — O365 / SaaS posture O365

Checks Microsoft 365 configuration against CISA benchmarks.

ScubaGear Developed by CISA (US govt). Checks MFA enforcement, Teams external access, Exchange DKIM/DMARC, SharePoint sharing settings. PowerShell tool. Run monthly.
Pentest / Active scanning On-demand

Actively tries to find and exploit vulnerabilities. Triggered from dashboard UI.

kube-hunter Hunts for attack paths in AKS: exposed APIs, privilege escalation, service account token leaks. Run as on-demand job from UI.
Nuclei Template-based scanner. 10,000+ templates for APIs, web apps, cloud misconfigs. Fast, highly configurable. Run from UI against your endpoints.
03

System architecture — the full picture

Five layers. Each tool plugs into the normalizer. The dashboard talks only to the API. The API talks only to the database and job queue.

Key design principle: The dashboard never calls a scanner directly. It only talks to FastAPI. FastAPI reads from PostgreSQL (for findings) or writes to Redis (for on-demand jobs). This keeps everything decoupled — you can add a new tool by writing one normalizer adapter, nothing else changes.
04

Two data flows — scheduled vs on-demand

Every tool falls into one of two flows. Understanding this makes the whole system make sense.

Scheduled flow (most tools)Continuous
01 Kubernetes CronJob fires (e.g. every 6 hours)
02 Scanner pod spins up (Prowler, Kubescape, etc.)
03 Scanner runs, outputs JSON to shared volume or stdout
04 Normalizer sidecar/init container picks up JSON
05 Normalizer writes findings to PostgreSQL
06 Dashboard shows new data on next refresh

Tools: Prowler, Kubescape, ScubaGear, Gitleaks, TruffleHog, kube-bench, AzureHound

On-demand flow (pentest tools)UI-triggered
01 User clicks "Run kube-hunter" in dashboard UI
02 React → POST /api/trigger-scan { tool: "kube-hunter" }
03 FastAPI pushes job to Redis queue
04 Celery worker picks job, creates Kubernetes Job pod
05 Scanner pod runs, output → normalizer → PostgreSQL
06 WebSocket pushes "scan complete" → UI shows report

Tools: kube-hunter, Nuclei, trivy cluster scan, ScubaGear (on-demand)

Falco is different from both — it's always-on as a DaemonSet, pushing alerts in real-time to a webhook endpoint on your FastAPI. FastAPI writes them to PostgreSQL as findings with severity=CRITICAL. The dashboard shows them instantly via WebSocket.
05

The common findings schema — design this first

This is the most important design decision. Every tool adapter maps to this schema. Get this right and everything else is just mapping exercise.

PostgreSQL — findings tableSQL
CREATE TABLE findings (
  id            UUID           PRIMARY KEY DEFAULT gen_random_uuid(),
  scan_id       UUID           REFERENCES scans(id),

  -- Which tool found this
  tool          TEXT           NOT NULL,  -- prowler | kubescape | falco | trivy | gitleaks | ...
  domain        TEXT           NOT NULL,  -- cspm | kspm | cwpp | ciem | image | iac | secrets | sspm

  -- Severity standardised across all tools
  severity      TEXT           NOT NULL,  -- CRITICAL | HIGH | MEDIUM | LOW | INFO

  -- What is affected
  resource_type TEXT,          -- storage_account | pod | image | git_repo | ...
  resource_id   TEXT,          -- Azure resource ID or K8s resource name
  resource_name TEXT,
  region        TEXT,

  -- The finding itself
  check_id      TEXT,          -- e.g. prowler check ID: azure_storage_blob_public_access_disabled
  title         TEXT           NOT NULL,
  description   TEXT,
  remediation   TEXT,
  references    JSONB,         -- links to CIS benchmark, CVE, etc.

  -- Raw data from the tool (keep original for debugging)
  raw_output    JSONB,

  -- Status tracking
  status        TEXT           DEFAULT 'open',  -- open | acknowledged | suppressed | resolved
  first_seen    TIMESTAMPTZ    DEFAULT NOW(),
  last_seen     TIMESTAMPTZ    DEFAULT NOW(),
  resolved_at   TIMESTAMPTZ,

  -- Tags for filtering
  tags          TEXT[]         DEFAULT '{}'  -- e.g. ['cis-1.3', 'pci-dss', 'azure']
);

CREATE TABLE scans (
  id          UUID     PRIMARY KEY DEFAULT gen_random_uuid(),
  tool        TEXT     NOT NULL,
  trigger     TEXT,    -- scheduled | manual | cicd
  status      TEXT,    -- running | completed | failed
  started_at  TIMESTAMPTZ,
  finished_at TIMESTAMPTZ,
  findings_count JSONB -- { critical: 3, high: 12, medium: 45 }
);

Infrastructure setup — prerequisites

Everything runs inside your existing AKS cluster. You need one dedicated namespace and a few shared services.

Create the security namespace and core infrastructurebash
# Create dedicated namespace for the entire CNAPP stack
kubectl create namespace cnapp-security

# Create service account for scanners (needs read-only access to cluster)
kubectl create serviceaccount cnapp-scanner -n cnapp-security

# Create ClusterRole for read-only cluster access
kubectl apply -f - <<EOF
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cnapp-readonly
rules:
  - apiGroups: ["*"]
    resources: ["*"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: cnapp-readonly-binding
subjects:
  - kind: ServiceAccount
    name: cnapp-scanner
    namespace: cnapp-security
roleRef:
  kind: ClusterRole
  name: cnapp-readonly
  apiGroup: rbac.authorization.k8s.io
EOF

# Deploy PostgreSQL (use Azure Database for PostgreSQL in prod)
helm repo add bitnami https://charts.bitnami.com/bitnami
helm install cnapp-postgres bitnami/postgresql \
  --namespace cnapp-security \
  --set auth.postgresPassword=changeme \
  --set auth.database=cnapp \
  --set primary.persistence.size=20Gi

# Deploy Redis for job queue
helm install cnapp-redis bitnami/redis \
  --namespace cnapp-security \
  --set auth.password=changeme \
  --set replica.replicaCount=0  # single node for small teams

# Create secret with Azure credentials for scanners
kubectl create secret generic azure-creds \
  --namespace cnapp-security \
  --from-literal=AZURE_CLIENT_ID=<your-client-id> \
  --from-literal=AZURE_CLIENT_SECRET=<your-client-secret> \
  --from-literal=AZURE_TENANT_ID=<your-tenant-id> \
  --from-literal=AZURE_SUBSCRIPTION_ID=<your-subscription-id>
Azure identity for scanners: Create a dedicated App Registration in Entra ID. Assign it Security Reader + Reader roles on your subscription. Never use your own credentials. Store the client secret in Azure Key Vault and reference it via AKS managed identity / CSI driver.

Installing CSPM tools — Prowler

Prowler is your primary CSPM tool. It runs as a Kubernetes CronJob, outputs JSON, and your normalizer picks it up.

Prowler CronJob — runs every 6 hoursyaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: prowler-cspm
  namespace: cnapp-security
spec:
  schedule: "0 */6 * * *"   # every 6 hours
  concurrencyPolicy: Forbid
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: cnapp-scanner
          restartPolicy: Never
          initContainers:
            - name: prowler
              image: toniblyx/prowler:latest
              env:
                - name: AZURE_CLIENT_ID
                  valueFrom:
                    secretKeyRef: { name: azure-creds, key: AZURE_CLIENT_ID }
                - name: AZURE_CLIENT_SECRET
                  valueFrom:
                    secretKeyRef: { name: azure-creds, key: AZURE_CLIENT_SECRET }
                - name: AZURE_TENANT_ID
                  valueFrom:
                    secretKeyRef: { name: azure-creds, key: AZURE_TENANT_ID }
                - name: AZURE_SUBSCRIPTION_ID
                  valueFrom:
                    secretKeyRef: { name: azure-creds, key: AZURE_SUBSCRIPTION_ID }
              command: ["prowler", "azure",
                        "-M", "json",
                        "-o", "/output/prowler-results.json"]
              volumeMounts:
                - name: output
                  mountPath: /output
          containers:
            - name: normalizer
              image: your-registry/cnapp-normalizer:latest
              env:
                - name: DATABASE_URL
                  value: "postgresql://postgres:changeme@cnapp-postgres/cnapp"
                - name: TOOL
                  value: "prowler"
                - name: INPUT_FILE
                  value: "/output/prowler-results.json"
              volumeMounts:
                - name: output
                  mountPath: /output
          volumes:
            - name: output
              emptyDir: {}
Prowler normalizer — maps to common schema (Python)python
# normalizers/prowler.py
import json, uuid
from datetime import datetime, timezone

SEVERITY_MAP = {
    "critical": "CRITICAL",
    "high":     "HIGH",
    "medium":   "MEDIUM",
    "low":      "LOW",
    "informational": "INFO"
}

def normalize_prowler(raw: dict, scan_id: str) -> dict:
    return {
        "id":            str(uuid.uuid4()),
        "scan_id":       scan_id,
        "tool":          "prowler",
        "domain":        "cspm",
        "severity":      SEVERITY_MAP.get(raw.get("Severity", "").lower(), "INFO"),
        "resource_type": raw.get("ResourceType"),
        "resource_id":   raw.get("ResourceId"),
        "resource_name": raw.get("ResourceName"),
        "region":        raw.get("Region"),
        "check_id":      raw.get("CheckId"),
        "title":         raw.get("CheckTitle"),
        "description":   raw.get("Description"),
        "remediation":   raw.get("Remediation", {}).get("Recommendation", {}).get("Text"),
        "status":        "open"  if raw.get("Status") == "FAIL"  else "resolved",
        "raw_output":    raw,
        "first_seen":    datetime.now(timezone.utc).isoformat(),
        "last_seen":     datetime.now(timezone.utc).isoformat(),
        "tags":          ["azure", "cspm"] + raw.get("Compliance", [])
    }

FastAPI webhook to receive Falco alertspython
# api/routes/webhooks.py
from fastapi import APIRouter, BackgroundTasks
from .normalizers.falco import normalize_falco
from .database import insert_finding
from .websocket import broadcast_alert

router = APIRouter()

@router.post("/webhook/falco")
async def receive_falco_alert(payload: dict, background_tasks: BackgroundTasks):
    finding = normalize_falco(payload)
    background_tasks.add_task(insert_finding, finding)
    background_tasks.add_task(broadcast_alert, finding)  # push to WebSocket
    return {"status": "received"}

# normalizers/falco.py
def normalize_falco(raw: dict) -> dict:
    PRIORITY_MAP = {
        "EMERGENCY": "CRITICAL", "ALERT": "CRITICAL",
        "CRITICAL":  "CRITICAL", "ERROR":    "HIGH",
        "WARNING":   "HIGH",     "NOTICE":   "MEDIUM",
        "INFO":      "LOW",      "DEBUG":    "INFO"
    }
    output_fields = raw.get("output_fields", {})
    return {
        "tool":          "falco",
        "domain":        "cwpp",
        "severity":      PRIORITY_MAP.get(raw.get("priority", "INFO"), "INFO"),
        "resource_type": "container",
        "resource_name": output_fields.get("container.name"),
        "resource_id":   output_fields.get("k8s.pod.name"),
        "title":         raw.get("rule"),
        "description":   raw.get("output"),
        "raw_output":    raw,
        "tags":          ["runtime", "cwpp", "realtime"]
    }

Pentest runner — UI-triggered scans

This is what makes your dashboard different from just a reporting tool. Users click a button, a scan runs, results appear in the UI.

Celery worker — creates K8s Job for each on-demand scanpython
# workers/scanner_worker.py
from celery import Celery
from kubernetes import client, config
import uuid, json

app = Celery("cnapp", broker="redis://cnapp-redis:6379/0")

SCANNER_IMAGES = {
    "kube-hunter": "aquasec/kube-hunter:latest",
    "nuclei":      "projectdiscovery/nuclei:latest",
    "trivy-cluster": "aquasec/trivy:latest",
}

def build_scanner_command(tool: str, params: dict) -> list:
    if tool == "kube-hunter":
        return ["kube-hunter", "--pod", "--report", "json"]
    elif tool == "nuclei":
        target = params.get("target", "https://your-app.com")
        return ["nuclei", "-u", target, "-json", "-o", "/output/nuclei.json"]

@app.task
def run_scanner(tool: str, scan_id: str, params: dict = {}):
    config.load_incluster_config()
    batch_v1 = client.BatchV1Api()

    job_name = f"cnapp-scan-{tool}-{scan_id[:8]}"
    job = client.V1Job(
        metadata=client.V1ObjectMeta(name=job_name, namespace="cnapp-security"),
        spec=client.V1JobSpec(
            ttl_seconds_after_finished=3600,
            template=client.V1PodTemplateSpec(
                spec=client.V1PodSpec(
                    service_account_name="cnapp-scanner",
                    restart_policy="Never",
                    containers=[client.V1Container(
                        name="scanner",
                        image=SCANNER_IMAGES[tool],
                        command=build_scanner_command(tool, params),
                        env=[
                            client.V1EnvVar(name="SCAN_ID", value=scan_id),
                            client.V1EnvVar(name="DB_URL",
                                value_from=client.V1EnvVarSource(
                                    secret_key_ref=client.V1SecretKeySelector(
                                        name="cnapp-secrets", key="database-url")
                                ))
                        ]
                    )]
                )
            )
        )
    )
    batch_v1.create_namespaced_job("cnapp-security", job)
    return {"job_name": job_name, "scan_id": scan_id}
FastAPI — trigger scan endpoint + WebSocketpython
# api/routes/scans.py
from fastapi import APIRouter, WebSocket
from workers.scanner_worker import run_scanner
import uuid

router = APIRouter()
connected_clients = []

@router.post("/api/trigger-scan")
async def trigger_scan(body: dict):
    scan_id = str(uuid.uuid4())
    tool    = body["tool"]   # kube-hunter | nuclei | trivy-cluster
    params  = body.get("params", {})

    # Push to Celery queue
    run_scanner.delay(tool, scan_id, params)

    return {"scan_id": scan_id, "status": "queued"}

@router.websocket("/ws/alerts")
async def websocket_endpoint(ws: WebSocket):
    await ws.accept()
    connected_clients.append(ws)
    try:
        while True:
            await ws.receive_text()  # keep alive
    finally:
        connected_clients.remove(ws)

async def broadcast_alert(finding: dict):
    for ws in connected_clients:
        await ws.send_json(finding)

React dashboard — what to build

Five screens. Build them in this order — each is useful on its own.

1.
Posture overview (home screen)
Risk score (0-100), findings by severity (donut chart), findings by domain (bar chart), trend over 30 days (line chart). This is what management looks at. Stack: React + Recharts.
2.
Findings table
All findings from all tools in one table. Filter by: domain, severity, tool, status, resource type. Search by resource name. Click a finding → detail panel with description, remediation steps, raw output. Mark as acknowledged / suppressed.
3.
Pen-test runner
List of available scanners (kube-hunter, Nuclei, etc.) with description. "Run now" button → POST /api/trigger-scan → job status polling → results appear as findings AND as formatted report in modal. WebSocket shows "scan complete" notification.
4.
Cluster / connection management
Add AKS clusters, Azure subscriptions, git repos. Configure scanner credentials. Enable/disable specific tools. Set scan schedules. This is the "add your infrastructure" screen — like Wiz's onboarding.
5.
Alerts + notification settings
Real-time alerts from Falco via WebSocket. Notification rules: "if severity=CRITICAL and domain=cwpp, send Slack alert immediately." Configure Slack/Teams webhooks, email. Alert history log.
